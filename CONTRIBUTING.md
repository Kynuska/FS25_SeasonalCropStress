# Contributing to FS25_SeasonalCropStress

Thanks for your interest in contributing. This document covers everything you need to know before opening an issue or submitting a pull request.

---

## Before You Start

Read `CLAUDE.md` — it is the authoritative reference for this project's architecture, known FS25 Lua constraints, and code patterns. Anything in that file takes precedence over general Lua or modding conventions you may know from other projects.

Read `DEVELOPMENT.md` for environment setup, build steps, and how to run the mod locally.

---

## Reporting Bugs

Use the **Bug Report** issue template. It requires a log snippet — issues opened without one will be closed immediately. The game log is at:

```
%USERPROFILE%\Documents\My Games\FarmingSimulator2025\log.txt
```

Filter for lines tagged `[CropStress]`. Include everything from the first `[CropStress]` line on load through the point where the problem occurs.

---

## Requesting Features

Use the **Feature Request** issue template. Before submitting, check the roadmap in `README.md` — your idea may already be planned.

Keep feature requests scoped. A request for "improve irrigation" will be closed. A request for "allow drip line coverage to be adjusted after placement" is actionable.

---

## Reporting Mod Compatibility Issues

Use the **Compatibility Report** issue template. Include the exact mod name, version, and where the conflict occurs (log error, wrong behavior, crash). Vague reports like "doesn't work with X" without reproduction steps will be closed.

## Submitting a Pull Request

### Setup

1. Fork the repository and clone your fork.
2. Work on a branch named after what you are changing: `fix/hud-moisture-display`, `feat/drip-line-rotation`, etc.
3. Build with `bash build.sh --deploy` and verify your changes in-game before opening a PR.

### Code Standards

- **Lua 5.1 only.** No `goto`, no `continue`, no `os.time()`, no `os.date()`. See the "What DOESN'T Work" table in `CLAUDE.md` for the full list.
- Match the existing style — indentation, naming conventions, comment format.
- Do not add docstrings, type annotations, or comments to code you did not change.
- Do not refactor surrounding code when fixing a bug. Change only what is necessary.
- Do not add error handling for scenarios that cannot happen in normal operation.
- All new files must be added to `main.lua` in the correct load phase order.
- No file may exceed 1500 lines. If your changes push a file over that limit, split it as described in `CLAUDE.md`.

### Patterns to Follow

- Cross-mod globals must be read via `getfenv(0)["globalName"]` — plain global access only sees the current mod's environment.
- New subsystems follow the `initialize() / hourlyUpdate() / delete()` lifecycle.
- Event-based communication between subsystems uses `CropEventBus`, not `g_messageCenter`.
- GUI dialogs extend `MessageDialog`, use the XML structure from `TakeLoanDialog.xml`, and always call `superClass().onOpen(self)` in `onOpen()`.
- HUD rendering uses `renderOverlay` with a shared `createImageOverlay` handle — never `drawFilledRect`.

### Pull Request Checklist

- [ ] `bash build.sh --deploy` completes with zero errors
- [ ] Tested in-game — describe what you tested in the PR description
- [ ] No files exceed 1500 lines
- [ ] No new globals added without a corresponding `getfenv(0)["name"] = value` export
- [ ] No branding added (no "Generated by", no AI references, no third-party tool credits)
- [ ] `CLAUDE.md` updated if you discovered a new FS25 constraint or confirmed an unverified API

### What Gets Rejected

- PRs that add features outside the current phase scope without prior discussion
- PRs that reformat or refactor code not related to the stated change
- PRs without in-game testing evidence
- PRs that introduce any of the broken patterns documented in `CLAUDE.md`
